  EXECUTIVE SUMMARY
================================================================================

This report documents a full macOS restoration operation performed on a
MacBook Pro (Late 2019, Intel, T2 Security Chip) using a Mac mini (M4 Pro,
Apple Silicon) as the host workstation. The operation was executed entirely
via command-line interface using Claude AI's Terminal service, leveraging
direct hardware access through Apple's Thunderbolt Target Disk Mode protocol.

The target MacBook Pro had suffered a broken Signed System Volume (SSV) seal —
a cryptographic integrity failure introduced in macOS Big Sur that prevented
the laptop from booting and blocked all standard recovery methods. Standard
approaches including Cmd+R Recovery and Internet Recovery were non-functional.

The operation was completed successfully, resulting in a bootable macOS
Sequoia 15.7.7 installer written to the MacBook Pro's internal SSD, ready
for a fresh installation that would automatically restore a valid SSV seal.

================================================================================
  2. PROBLEM STATEMENT
================================================================================

  2.1 THE SSV SEAL FAILURE
  ─────────────────────────────────────────────────────────────────────────────
  Starting with macOS Big Sur (11.0), Apple introduced the Signed System
  Volume (SSV) — a read-only, cryptographically sealed copy of the macOS
  system volume. At boot, the T2 chip verifies the seal by computing a hash
  tree over every file in the System volume. If ANY file differs from the
  sealed state, the Mac refuses to boot.

  The MacBook Pro's SSV seal was broken. Root causes can include:
    • Failed or interrupted macOS upgrade
    • Third-party kernel extension or system modification
    • Interrupted firmware update
    • Storage corruption or partial disk failure

  2.2 WHY STANDARD RECOVERY FAILED
  ─────────────────────────────────────────────────────────────────────────────
  On T2 Macs with a broken seal, all standard recovery paths fail:

  Cmd+R (macOS Recovery):
    The T2 chip validates the local recovery partition before loading it.
    A broken seal can prevent the Recovery environment from initializing,
    leaving the user at a black screen or spinning globe indefinitely.

  Option+Cmd+R (Internet Recovery):
    Not explicitly tested in this engagement. On T2 Macs, Apple's servers
    perform chip validation before serving the recovery image. Given the
    confirmed seal failure and Cmd+R non-functionality, Internet Recovery
    was assessed as likely non-viable and an external restore was initiated.

  Result: The MacBook Pro could not be recovered through any on-device
  mechanism. An external restore was required.

  2.3 WHY startosinstall --volume FAILED
  ─────────────────────────────────────────────────────────────────────────────
  The standard macOS command-line install tool (startosinstall) accepts a
  --volume flag to target a specific disk. However, this consistently
  failed when executed on the Apple Silicon Mac mini targeting the Intel
  MacBook Pro disk:

    $ sudo startosinstall --volume /Volumes/MVVJ2LL --agreetolicense
    [prints usage and exits]

  Investigation (via strings analysis of the binary) confirmed --volume IS
  a valid flag. The failure was due to a fundamental architectural restriction:

    • Apple Silicon Macs use a different security model (SPSB/LocalPolicy)
    • startosinstall on Apple Silicon cannot cross-install to Intel targets
    • The Intel disk's T2 security policy is not accessible from the host
    • Both --volume alone and --eraseinstall + --volume combinations fail
      (--eraseinstall and --volume are mutually exclusive by design)

  This is an undocumented but consistent behavior on macOS Sequoia/Tahoe
  running on Apple Silicon hardware.

================================================================================
  3. SOLUTION DESIGN
================================================================================

  The solution bypassed startosinstall entirely in favor of Apple's
  createinstallmedia tool, which has no architecture restriction and works
  from any Mac to any target disk regardless of chip generation.

  createinstallmedia writes a complete, Apple-signed, self-contained bootable
  macOS installer directly to the target disk. Because the installer is
  Apple-signed firmware, it passes T2 Secure Boot validation and allows the
  MacBook Pro to boot from its own internal disk.

  DISK STRATEGY:
  Rather than consuming the entire 500 GB with the installer (which would
  leave no room for macOS to install onto), the disk was split into two
  partitions post-createinstallmedia:

    Partition 1: HFS+  "Install macOS Sequoia"  22 GB  — Bootable installer
    Partition 2: APFS  "Macintosh HD"          477 GB  — Clean install target

  This allows the MacBook Pro to:
    (a) Boot from the installer partition (hold Option at startup)
    (b) Install macOS Sequoia to the APFS partition
    (c) Complete a full fresh install with a valid SSV seal

  KEY INSIGHT — HFS+ REQUIREMENT:
  createinstallmedia rejects APFS-formatted target volumes:
    "APFS disks may not be used as bootable install media."
  The disk must first be erased as HFS+ (Mac OS Extended, Journaled)
  before createinstallmedia will proceed. This is a critical step that is
  not documented prominently in Apple's official guidance.

================================================================================
  6. SSV SEAL RESTORATION
================================================================================

  The Signed System Volume seal is NOT manually repaired — it is created
  fresh during every macOS installation. When macOS Sequoia installs onto
  the "Macintosh HD" APFS volume, the installer:

    1. Writes the complete macOS system to the System volume
    2. Creates a read-only APFS snapshot of the System volume
    3. Computes a cryptographic hash tree over all system files
    4. Seals the snapshot with Apple's signing key
    5. Records the seal in the APFS volume metadata

  The T2 chip then validates this seal on every subsequent boot. A fresh
  installation always produces a valid, verifiable seal — this is why a
  clean reinstall is the definitive fix for SSV seal corruption.

  To verify seal integrity post-installation:
    $ diskutil info / | grep Sealed
    Sealed: Yes

================================================================================
  7. KEY TECHNICAL FINDINGS & LESSONS
================================================================================

  FINDING 1 — Architecture Restriction in startosinstall
  ─────────────────────────────────────────────────────────────────────────────
  startosinstall --volume fails silently (prints usage, exits 1) when run
  on Apple Silicon attempting to target an Intel Mac's disk in TDM. This is
  an undocumented restriction. The binary DOES contain the --volume flag
  (confirmed via strings analysis) but runtime validation rejects cross-
  architecture installs. The only working alternative is createinstallmedia.

  FINDING 2 — createinstallmedia Requires HFS+
  ─────────────────────────────────────────────────────────────────────────────
  createinstallmedia refuses APFS-formatted volumes:
    "APFS disks may not be used as bootable install media."
  Any target disk must be erased as JHFS+ (Mac OS Extended, Journaled)
  before createinstallmedia will proceed. This applies even when the target
  is an Intel Mac's internal SSD accessed via TDM.

  FINDING 3 — splitPartition Erases Content
  ─────────────────────────────────────────────────────────────────────────────
  diskutil splitPartition reformats the source partition (disk4s2), erasing
  the installer content that was just written. The installer must be re-run
  against the smaller post-split partition. Scripts should account for this
  by treating split and createinstallmedia as a two-step sequence.

  FINDING 4 — Thunderbolt TDM Disk Identity & Identifier Reassignment
  ─────────────────────────────────────────────────────────────────────────────
  When an Intel Mac connects in TDM, its default APFS volume name reflects
  the machine's model number (e.g., "MVVJ2LL" = MacBook Pro 16-inch 2019
  Space Gray, model MVVJ2LL/A — NOT the serial number). This can be used to
  confirm correct disk identification before destructive operations.
  Additionally, macOS may reassign the disk identifier (e.g., disk6 → disk4)
  between Thunderbolt disconnect/reconnect cycles. Scripts must re-run
  diskutil list after any reconnection to confirm the current identifier.

  FINDING 5 — T2 Firmware Password Cannot Be Removed via TDM
  ─────────────────────────────────────────────────────────────────────────────
  If a T2 firmware password is set on the target Mac, it will manifest as
  a lock icon in the boot picker regardless of disk state. The firmware
  password is stored in the T2 chip's secure enclave — not on the disk —
  and CANNOT be removed or bypassed via Target Disk Mode. Remediation
  requires Apple Configurator 2 in DFU mode.

================================================================================
  8. AUTOMATION SCRIPT
================================================================================

  The operation has been fully automated in:
    macos_restore_tdm.sh
    https://github.com/IT-Solutions-USA/mac-admin/blob/main/mac-admin/macos_restore_tdm.sh

  The script performs all phases in sequence with error checking:
    1. Download macOS installer (if not present)
    2. Validate target disk exists and is external
    3. Erase disk as HFS+ (JHFS+)
    4. Run createinstallmedia
    5. Split partition (22 GB installer + 477 GB APFS target)
    6. Re-run createinstallmedia on smaller partition
    7. Eject disk
    8. Print next-step instructions

  Configuration variables at the top of the script allow customization of:
    • INSTALLER_APP   — path to macOS installer
    • TARGET_DISK     — disk identifier (e.g., disk4, disk6)
    • INSTALLER_PART_SIZE — installer partition size (default 22g)
    • MACOS_VERSION   — version to download if installer missing

  9. PREREQUISITES FOR FUTURE OPERATIONS
================================================================================

  To replicate this operation, the following are required:

  Hardware:
    ✓ Any Mac running macOS Monterey 12.0 or later as host
    ✓ Target Intel Mac with T2 chip (MacBook Pro 2016–2019, etc.)
    ✓ Thunderbolt 3 or Thunderbolt 4 cable

  Software:
    ✓ macOS installer app in /Applications (or internet access to download)
    ✓ sudo privileges on host Mac
    ✓ Sufficient free disk space (~20 GB) for installer download

  Procedure:
    1. Boot target Mac holding T → confirm Thunderbolt symbol on screen
    2. Connect Thunderbolt cable to host Mac
    3. Confirm disk appears: diskutil list | grep external
    4. Set TARGET_DISK in macos_restore_tdm.sh
    5. Run: sudo ./macos_restore_tdm.sh
    6. When complete: disconnect cable, boot target holding Option (⌥)
    7. Select "Install macOS [version]" → install to "Macintosh HD"
